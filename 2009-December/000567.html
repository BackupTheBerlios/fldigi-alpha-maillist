To All,

1. I got flwrap from the alpha page.

2. OK on it being a compressed file and not encrypted.
  A. MARS is directed by the Department of Defense not the FCC.
      The DOD says we do not use encryption in MARS unless the client's
message is encrypted. Someone would have to run FLWRAP through the DOD for
authorisation for MARS use.

3. Having the message sent in plain language is what we need.
  A. All MARS messages are text files.
  B. If for some reason we need to send any other type of file we
would probably use EasyPal.
  C. .csv files could also be sent by fldigi.

4. Trying to get all MARS members to use fldigi would require
an edict from the Chief's of each MARS service to declare it the
only software to use. Until that time we have MixW, DM-780 and
others in use. Most of our transmissions are in MT-63 1K or 2K
long interleaf. Occasionally we use Olivia.
  A. FLWRAP or WRAP will work with the other software but they
don't have the drag and drop properties of fldigi.
  B. Getting everyone to use FLWRAP would also require an edict.
  C. The best and easiest way to get everyone to use it is to make
it as simple and universal as possible.
  D. In a MARS net there may be 15 or 20 members getting the
traffic, unlike an EMCOMM situation in a non-MARS net where
the stations would probably be one on one. As an EC  and EMA
radio officer for my county I would use the FLWRAP.

5. The old WRAP did not have any objections from the upper
echelon that I am aware of; even though it added several extra
lines to the message (not military format).
  A. If the old WRAP will work, that's all we need.

6. How much time is saved by compressing a 500 byte text file?
  A. MARS has the bandwidth to use MT63-2K; if there are faster modes
we would like to try them.

7. It would be tough to teach our older MARS members to use command
line switches; even if I could pry them away from MixW that they paid $50 for.

8. Where can I send a screen shot of how I set up fldigi and the message
folders? I think if you try it you will find it very fast and efficient.

Thanks for all the suggestions and the help, I really appreciate it.

> The purpose of compressing all files was to reduce the transmission time.
> Files transferred using flwrap contain data which must be verified.  If you
> are simply eyeballing the internal contents of the wrapped file you are
> defeating its purpose.  Why waste time with the headers etc.  Just transmit
> the original file and accept propagation errors.  If the data must be intact
> and verifiable then it makes no difference whether you can read the data

> data is useless.  A good example might be a spreadsheet in comma separated
> value (.CSV) format that contains numeric data.  How would you verify this
> data without the wrap technology.  You would either have to repeat the file
