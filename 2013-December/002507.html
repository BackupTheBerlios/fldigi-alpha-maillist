Hi Remi
I have created synthetic Morse code test audio files with SNR from +60 dB to -20 dB and with various speeds up to 40 WPM.

I experimented also adding some QSB on these synthetic files but I haven't figured out the right formula to create realistic ionospheric effects. There is a Windows program (Pathsim or something like that) that I used earlier but the results did not sound very realistic. As expected decoding accuracy falls quickly when SNR falls below threshold in these simulated audio files.

I have also thought about adding random fluctuations on Morse timing but my Morse generator is Octave based and haven't had time to start rebuilding that yet. Humans seem to have fairly individualistic rhythm patterns that don't follow Gaussian distribution except when aggregated across multiple individuals. Modeling those individual patterns may be difficult....

I was quite surprised to get on "Ask Slashdot" home page with my question. I got some good ideas but also many replies with no new information. 

This pursuit of improved CW decoder is quite challenging. Not just from software perspective but also in getting good test material and actionable feedback. I have certainly learned a lot but I feel that I am not making much progress.

73 and happy holidays 

Mauri AG1LE

On Wed, Dec 25, 2013 at 4:27 AM, mauri at innomore.com <mauri at innomore.com> wrote:

I have been working on a new Bayesian Morse decoder for FLDIGI for a while.  
The goal was to have a CW decoder that adapts better to different operators rhytms, sudden speed changes, signal fluctuations and noise and still is able to decode Morse code fairly accurately.  

While this software is not perfect and has some bugs I am happy to release this brand new version for alpha testing.  I finished the software integration to FLDIGI v3.21.75 and spent few hours today listening bands.  Below is a small example of W1AW code bulletin at 3.58105 MHz  received with KX3 and a side by side comparison of legacy CW decoder  vs  Bayesian decoder:  
http://ag1le.blogspot.com/2013/12/new-morse-decoder-part-2.html 

The full source code is available in here:  http://www.w1hkj.com/ag1le 

I would like to get some alpha testers who are not afraid of compiling this code from sources,  playing with it, breaking it and hopefully also helping me to find the bugs.  
For example  - the noise level estimator (noise.c) needs some serious work.  If you have a algorithm that does the job better this would be one place to start improving.  Also, if you know Viterbi or Kalman filter algorithms like your pockets I would like to get help fixing a bug that appears in posterior probability calculations in probp.c occasionally.  

Anyways,  happy holidays for all FLDIGI users and hope to work you over the radio.
