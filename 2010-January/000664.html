>> If you left click in the Rx widget (that's Linux-speak for control)
>> the cursor and the keyboard focus is assigned to that area of the
>> gui.  The Rx widget rejects all keyboard entries.  So you can type
>> away and get nothing.  Pressing the keyboard ENTER key will restore
>> keyboard focus to the Tx widget.  Left clicking in the Tx widget will
>> also do the same (be sure that the insert cursor is at the end of the
>> text).  The TAB key moves the cursor from the current TX position to
>> the end of text and reverse.  This is outlined in the on-line help.
>>
>> So I will risk saying that there is a high probability that you had
>> just transferred data from the Rx widget to the logging widgets. 
>> Perhaps your last mouse click in the Rx area was a left click.  That
>> would cause the effect reported by several testers.  It would also
>> explain the seemingly random nature of the observation.

This one has been difficult to find because it happened only randomly it
seemed.  But I finally found the trigger that causes it.  When I take
the mouse and left click into the blue transmit pane, everything works
fine.  However if the mouse moves down into the waterfall and brings up
the yellow freq markers, even though the left button has not been
pressed to move the frequency, the caps stop, but the lowercase keep
going.  I think this happens to me when I use the mouse to click on the
transmit pane, and then use my elbow to slide it out of the way of the
keyboard so I can type.  My desk is small, and I share the space between
the two, so the randomness came from where the mouse landed.  I think
this may be a bug, because once I press a left button for the cursor, it
should stay active, no matter where the mouse rolls to.  Anyway an
answer to a frustrating search for the problem.
